# Подвійна оптимізація документації: для людей та ШІ

Сучасна Open Source документація стоїть перед унікальним викликом — одночасно задовольняти потреби людей-розробників і ШІ-асистентів. Проект Beads від Steve Yegge демонструє як переваги, так і критичні проблеми AI-генерованої документації: понад 130 тисяч рядків коду, створених за 6 днів «вайб-кодингу», супроводжуються документацією, яку спільнота називає «майже непроникною». Цей звіт пропонує практичну стратегію створення документації, що працює для обох аудиторій.

## Beads: технічна суть та проблема документації

**Beads** — це розподілений граф-трекер задач на базі Git, створений для AI-кодинг агентів. Проект вирішує критичну проблему «амнезії контекстного вікна» — коли AI-асистенти втрачають усе розуміння проекту між сесіями. Архітектура поєднує **JSONL-файли** як джерело істини, **SQLite** для швидких локальних запитів (до 100 мс), та **Git** для синхронізації між машинами.

Репозиторій **joyshmitz/beads** є форком оригінального steveyegge/beads з гілкою `docs/docusaurus-site`, що створює веб-документацію на базі Docusaurus. Оригінальний проект містить понад 10 markdown-файлів: INSTALLING.md, QUICKSTART.md, AGENT_MAIL.md, FAQ.md, TROUBLESHOOTING.md та інші. Однак попри цю кількість, спільнота висловлює серйозне невдоволення якістю контенту.

## Анатомія провалу: issue #376 та анти-патерни AI-документації

Issue #376 «I want to love Beads but the AI generated docs make it impossible» отримала **14 реакцій підтримки** та виявила системні проблеми AI-генерованої документації.

**Ключові анти-патерни, що критикуються спільнотою:**

| Анти-патерн | Опис |
|-------------|------|
| Гіперболізовані твердження | Маркетингова мова замість технічної точності |
| «Пухнаста» AI-документація | Відсутність глибини, яку дає людське авторство |
| Відсутня архітектура | Немає пояснення взаємодії Git/JSON/SQLite |
| Немає відновлення | Відсутні процедури виходу з помилкових станів |
| Непроникна структура | Обсяг AI-контенту ускладнює пошук авторитетної інформації |

Автор issue @srobertson зазначає: «Я можу розпізнати Claude-style автогенеровані доки, коли їх бачу. Сміливі, перебільшені твердження заважають вирішувати реальні баги». Головний запит спільноти — **хоча б один документ, відредагований людиною**, що пояснює модель даних.

Сам Steve Yegge визнав у блозі: «Це кострубата архітектура (за до-AI стандартами), що вимагає AI для обходу всіх edge cases». Це підтверджує критику: документація не може компенсувати архітектурну складність без людської курації.

## Специфікація llms.txt: оптимізація для ШІ

Стандарт **llms.txt**, запропонований Jeremy Howard у вересні 2024 року, вирішує проблему обмеженого контекстного вікна LLM при роботі з вебсайтами.

**Формат файлу** (Markdown):
```markdown
# Назва проекту
> Короткий опис ключових можливостей

Детальна інформація про проект у кількох абзацах.

## Документація
- [Швидкий старт](URL): Опис що містить документ
- [API Reference](URL): Технічна специфікація

## Optional
- [Changelog](URL): Історія змін
```

**Технічні вимоги:** файл розміщується за адресою `yoursite.com/llms.txt`, формат UTF-8, MIME-тип text/plain, рекомендований розмір до 10 КБ, обов'язковий H1-заголовок. Багато проектів також надають **llms-full.txt** — повний текст документації для прямого завантаження в LLM.

**Приклади успішних реалізацій:** Cloudflare організовує llms.txt за сервісами (Workers, AI Gateway), Stripe — за продуктами (Checkout, Payments, Billing), Zapier — за API-ендпоінтами. Спільний патерн: **10-20 ключових сторінок** з описами, чітка ієрархія, секція Optional для другорядного контенту.

## UX документації для людей: принципи читабельності

Оптимізація для людей принципово відрізняється від оптимізації для ШІ. Людям потрібні елементи, які ШІ не використовує або втрачає при обробці.

**Когнітивне навантаження** — ключова метрика людського UX. Робоча пам'ять утримує лише **3-7 елементів** одночасно. Принципи зменшення навантаження: видалення зайвої інформації (coherence), розбиття на порції (chunking), утримання пов'язаної інформації разом (contiguity).

**Типографічна ієрархія** будується на трьох рівнях: заголовки (привертають увагу), підзаголовки (структурують), основний текст (максимальна читабельність). Контраст розмірів і ваги шрифтів має бути очевидним — тонкі відмінності залишаються непоміченими.

**Навігація** для людей спирається на візуальні підказки: глобальна навігація присутня на всіх сторінках, локальна — контекстна для поточного розділу, breadcrumbs показують позицію в ієрархії, бічна панель забезпечує швидкий доступ до розділів.

| Аспект | Для людей | Для ШІ |
|--------|-----------|--------|
| Стиль читання | Лінійна нарація, сканування | Дискретні чанки, pattern matching |
| Контекст | Може виводити неявну інформацію | Працює лише з явно задокументованим |
| Візуальна інформація | Інтерпретує діаграми, layout | Втрачає значення без текстових альтернатив |
| Навігація | Візуальна ієрархія | Семантична структура, метадані |

**Практичний висновок:** документація, оптимізована для ШІ — це по суті відмінна документація: чітка, структурована, експліцитна. Ті самі принципи працюють для accessibility та screen readers.

## Методології технічного письма: Diátaxis та Google

**Фреймворк Diátaxis** (Daniele Procida, Canonical) визначає чотири типи документації за потребами користувачів:

**Tutorials** (навчання) — покрокові уроки для новачків, будують базові навички через практику. Формула: вступ → передумови → кроки → підсумок → наступні кроки.

**How-to Guides** (вирішення проблем) — практичні інструкції для конкретних завдань. Користувач уже знає основи, потрібно завершити задачу.

**Reference** (інформація) — технічні специфікації, точні описи параметрів, API, конфігурацій. Формат довідника.

**Explanation** (розуміння) — концептуальний контекст, фонова інформація, зв'язки між компонентами. «Чому» замість «як».

**Ключовий принцип:** кожен документ служить ОДНІЙ меті. Змішування типів (наприклад, tutorial з reference) погіршує обидва.

**Google Technical Writing** акцентує на: активний стан («Install the package» замість «The package can be installed»), короткі речення (одне речення = одна ідея), уникнення прислівників і прикметників (використовувати точні, вимірювані характеристики), визначення аудиторії (уникати «прокляття знання»).

**Спільні патерни успішних проектів** (React, Vue, Python, Rust): Getting Started для швидкого входу, Core Concepts для розуміння, API Reference для технічних деталей, Examples для практики, FAQ/Troubleshooting для проблем, Contributing для спільноти. Усі використовують версіонування документації та інтерактивні приклади коду.

### Специфікація llms.txt для Beads

```markdown
# Beads

> Distributed git-backed graph issue tracker for AI coding agents. 
> Solves context window amnesia across agent sessions.

Beads stores issues as JSONL in .beads/ directory, synced via git.
Local SQLite caches provide fast queries (<100ms). Four dependency 
types: blocks, related, parent-child, discovered-from.

## Getting Started
- [Installation](https://joyshmitz.github.io/beads/docs/installing): Platform-specific setup for macOS, Linux, Windows
- [Quick Start](https://joyshmitz.github.io/beads/docs/quickstart): Create first issue and integrate with AI agent

## Core Concepts
- [Architecture](https://joyshmitz.github.io/beads/docs/architecture): Git/JSON/SQLite data model interaction
- [Dependency Graph](https://joyshmitz.github.io/beads/docs/dependencies): How issues relate and block each other

## Reference
- [CLI Commands](https://joyshmitz.github.io/beads/docs/cli): Complete command reference with examples
- [Configuration](https://joyshmitz.github.io/beads/docs/config): All configuration options explained

## Optional

```

### Стратегія співіснування форматів

**Принцип:** один source of truth, два представлення.

1. **Markdown як джерело** — усі документи пишуться в Markdown з семантичною розміткою
2. **Docusaurus для людей** — рендерить Markdown у красивий сайт з навігацією, пошуком, темами
3. **llms.txt як індекс** — генерується автоматично або вручну оновлюється при змінах структури
4. **llms-full.txt** — конкатенація ключових документів для прямого завантаження в LLM

**Технічна реалізація:** плагін `docusaurus-plugin-llms` автоматично генерує llms.txt з конфігурації сайту. Альтернатива — скрипт у CI/CD, що збирає llms.txt з frontmatter метаданих документів.

**Критичний принцип для уникнення анти-патернів:** кожен документ МУСИТЬ проходити людську редакцію. AI може генерувати чернетки, але фінальна версія — результат людської курації. Це вирішує головну скаргу з issue #376.

## Висновок

Проект Beads ілюструє парадокс AI-ери: інструмент, створений для покращення роботи AI-агентів, страждає від документації, яку AI не може ефективно використовувати. Рішення — не відмова від AI у створенні документації, а **гібридний підхід**: AI генерує чернетки, людина курує, структурує та забезпечує архітектурну ясність.

llms.txt і людська документація не конкурують — вони доповнюють одне одного. llms.txt — це індекс для машин, Docusaurus — інтерфейс для людей. Обидва базуються на одному Markdown-контенті, але оптимізовані для різних споживачів.

Головний урок issue #376: технічна документація вимагає **авторитетного голосу**. AI може генерувати багато тексту, але лише людина може сказати «це архітектура, і ось чому вона працює саме так». Цей авторитет — те, чого найбільше бракує Beads і що можна виправити одним добре написаним документом.
