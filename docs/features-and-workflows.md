# Beads Features and Workflows

This doc combines Beads feature and workflow inventories under a single classification contract.

Sources of truth:
- Classification contract: `.sisyphus/evidence/task-2-classification-contract.md`
- Feature inventory evidence: `.sisyphus/evidence/task-7-beads-features.md`
- Workflow inventory evidence: `.sisyphus/evidence/task-8-beads-workflows.md`

Evidence paths inside inventories are kept repo-relative where possible (for example `beads/cmd/bd/main.go`).

## Glossary

- Evidence seed: A `.sisyphus/evidence/...` artifact that records extracted facts plus citations.
- Evidence path: A repo-relative pointer to code, tests, captured output, or evidence seed lines.
- Surface: A user-facing interface where interaction happens, for example a CLI command or HTTP route.
- Feature: A discrete capability exposed at a user-facing surface.
- Workflow: A repeatable end-to-end sequence that uses multiple steps or surfaces to accomplish an outcome.
- Status token: One of `implemented`, `partially_implemented`, `indicated`, `roadmap`, `deprecated`, `unknown/conflict`.
- Confidence: A qualitative strength label for the evidence supporting a claim.
- Cross-repo contract: A shared assumption across repos, often an env var, flag, path, or semantic.
- Conflict register: The place cross-repo contract mismatches are logged, using `unknown/conflict` when unresolved.
- Orchestrator: A process that coordinates another tool or repo, often by spawning it as a subprocess.

## Feature Classification Model

Terms (from `.sisyphus/evidence/task-2-classification-contract.md`):

- Feature: A discrete capability exposed at a user-facing surface (CLI command, HTTP route, file format, integration point). If it is one interaction at one surface, it is usually a feature.
- Workflow: A repeatable end-to-end sequence that orchestrates multiple steps or surfaces to accomplish an outcome. If it needs multiple interactions or multiple surfaces to describe accurately, it is usually a workflow.

Status taxonomy (locked tokens):

- `implemented`: User-visible behavior exists now. Requires non-doc evidence (code/tests/runtime proof). Docs-only evidence is never enough.
- `partially_implemented`: Some behavior exists but parts are stubbed, no-op, gated, or explicitly not yet implemented.
- `indicated`: Strong signals of intent exist, but implementation is not verified.
- `roadmap`: Future intent with no usable behavior today.
- `deprecated`: Exists or existed, but flagged as deprecated, replaced, hidden, or scheduled for removal.
- `unknown/conflict`: Evidence is contradictory or incomplete and cannot be resolved with available sources.

Confidence levels:

- High: At least one high-precedence non-doc source directly supports the claim.
- Medium: Evidence supports the claim but is indirect or missing a critical link.
- Low: Docs/design/TODO-only or weak signals.

Evidence precedence (higher wins on conflicts):

1. Code and handlers (implementation)
2. Executable surfaces (captured runtime proof)
3. Tests
4. Docs (reference)
5. Design docs (intent)
6. TODO/FIXME markers (intent signal)

Conflict handling:

- If two same-precedence sources disagree and you cannot prove which is correct, use `unknown/conflict` and record both claims and evidence.
- Do not downgrade to `partially_implemented` unless you have direct evidence of partial behavior.

Inventory row schemas (required fields):

- Features: surface, invocation, inputs, outputs, side effects, evidence path (plus recommended: feature, status, confidence, notes).
- Workflows: workflow, trigger, invocation, steps, surfaces touched, inputs, outputs, side effects, evidence path (plus recommended: status, confidence, preconditions, failure modes, idempotency, notes).

## Cross-Repo Workflow Reconciliation

- Source of truth for cross-repo interoperability evidence: `.sisyphus/evidence/task-13-cross-repo-workflow-reconciliation.md`.
- `gt` -> `bd` subprocess boundary: Beads expects correct routing to the intended `.beads` context via `BEADS_DIR` (preferred) or discovery fallbacks, so the caller must avoid leaking an inherited `BEADS_DIR` that targets the wrong database. Evidence: `beads/internal/beads/beads.go:267`, `beads/internal/beads/beads.go:276`, `gastown/internal/cmd/bd_helpers.go:54`.
- Formula interoperability under Gas Town: Beads formula search path includes `$GT_ROOT/.beads/formulas/` when `GT_ROOT` is set (orchestrator-provided formulas), which is how `gt` can drive formula/cook/mol workflows without copying formula files into every worktree. Evidence: `beads/cmd/bd/formula.go:32`, `beads/cmd/bd/formula.go:48`, `gastown/internal/cmd/sling_helpers.go:683`.
- Dolt server semantics under Gas Town: Beads uses `GT_ROOT` to share Dolt server state under `$GT_ROOT/.beads` and to disable auto-start when the server is daemon-managed; `gt` must pass `GT_ROOT` through to `bd` subprocesses to activate this mode. Evidence: `beads/internal/doltserver/doltserver.go:52`, `beads/internal/doltserver/doltserver.go:55`, `beads/cmd/bd/main.go:512`.
- Telemetry interoperability: Beads telemetry enables metrics export when `BD_OTEL_METRICS_URL` is set; `gt` mirrors its telemetry endpoints into `BD_OTEL_METRICS_URL` / `BD_OTEL_LOGS_URL` and sets `OTEL_RESOURCE_ATTRIBUTES` for subprocess inheritance. Evidence: `beads/internal/telemetry/telemetry.go:3`, `beads/internal/telemetry/telemetry.go:51`, `gastown/internal/telemetry/subprocess.go:52`.
- Key handoff variables (as evidenced): `BEADS_DIR`, `GT_ROOT`, `BD_DOLT_AUTO_COMMIT`, `BD_OTEL_METRICS_URL` / `BD_OTEL_LOGS_URL`, `OTEL_RESOURCE_ATTRIBUTES`. Evidence: `.sisyphus/evidence/task-13-cross-repo-workflow-reconciliation.md`.
- `unknown/conflict`: see `.sisyphus/evidence/task-13-cross-repo-workflow-reconciliation.md` for unresolved contradictions (dolt server pid path, auto-commit knob naming, and whether Beads telemetry honors `OTEL_RESOURCE_ATTRIBUTES`). Evidence: `.sisyphus/evidence/task-13-cross-repo-workflow-reconciliation.md`.

## Known Gaps / Indicated Only

These are explicitly called out gaps or contradictions that should be treated as `unknown/conflict` until resolved with higher-precedence evidence.

- Build-tag variant behavior for federation: the `bd federation` surface is build-artifact dependent. CGO builds (`//go:build cgo`) register operational subcommands (`sync`, `status`, peer management) with real store operations, while non-CGO builds (`//go:build !cgo`) provide a stub command that prints guidance and performs no peer sync/status work. Evidence: `.sisyphus/evidence/task-7-beads-features.md` (unknown/conflict row), `.sisyphus/evidence/task-8-beads-workflows.md` (rows and unknown/conflict), `beads/cmd/bd/federation.go`, `beads/cmd/bd/federation_nocgo.go`, `beads/.goreleaser.yml`.
- `noDbCommands` mismatch: root pre-run skip-db gating includes command names that do not appear as top-level Cobra `Use` literals (for example `sync`, `resolve-conflicts`). This makes it unclear whether the allowlist reflects legacy/compat behavior or stale mapping, and it can influence which invocations bypass store initialization. Evidence: `.sisyphus/evidence/task-7-beads-features.md` (unknown/conflict row), `.sisyphus/evidence/task-8-beads-workflows.md` (unknown/conflict bullets), `beads/cmd/bd/main.go`, `beads/cmd/bd/*.go`.

Evidence: `beads/internal/configfile/configfile.go` (comment and `GetDoltMode` body), `beads/cmd/bd/doctor/server.go`.
