# **Стратегічна архітектура документації гібридних людино-машинних систем: Глибоке дослідження екосистеми Beads та імплементація подвійних стандартів сприйняття**

## **Анотація**

Цей звіт представляє вичерпний аналіз стратегій документування для інструментарію нового покоління, орієнтованого на взаємодію людей та агентів штучного інтелекту. Об'єктом дослідження виступає проект **Beads** (форк docs/docusaurus-site), який стикається з фундаментальними викликами у забезпеченні прозорості архітектури та зручності використання. На основі критичного аналізу issue \#376, специфікацій llms.txt та фреймворку Diátaxis, розроблено детальну дорожню карту трансформації документації. Документ пропонує уніфіковану модель, що гармонізує потреби когнітивного сприйняття людини з вимогами контекстної обробки LLM, забезпечуючи при цьому технічну точність та операційну надійність.

**Розділ 1\. Еволюція інструментарію розробника: Від людино-центричних до агентно-орієнтованих систем**

### **1.1. Зміна парадигми в екосистемі DevTools**

Сучасна індустрія розробки програмного забезпечення перебуває на порозі тектонічного зсуву, який можна порівняти з появою систем контролю версій або впровадженням CI/CD. Традиційні інструменти — від IDE до трекерів задач — проектувалися з аксіоматичним припущенням, що кінцевим користувачем є людина. Інтерфейси, документація, повідомлення про помилки та архітектурні абстракції оптимізувалися під людський когнітивний апарат. Однак поява автономних агентів кодування (AI Coding Agents), таких як Devin, Claude Code, або кастомних рішень на базі LLM, змінює ландшафт.

Проект **Beads**, створений Стівом Єгге, є флагманом цієї нової хвилі. Він позиціонується як «оновлення пам'яті» (memory upgrade) для агентів, замінюючи неструктуровані плани в Markdown на структурований граф залежностей.1 На відміну від Jira чи Trello, Beads живе там, де живуть агенти — в коді, використовуючи Git як транспортний рівень та джерело істини.2 Це дозволяє агентам зберігати контекст довготривалих задач між сесіями, що є критичним для вирішення складних інженерних проблем.

Проте, ця інновація створює «долину смерті» у документації. Старі підходи до написання технічних інструкцій, розраховані на інтуїцію та візуальне сприйняття людини, не працюють для агентів, які мислять токенами та структурами. Водночас, документація, згенерована агентами для агентів (явище, що стає все більш поширеним), часто виявляється незрозумілою для людей-архітекторів, які повинні контролювати ці системи. Це призводить до кризи довіри та операційних ризиків, коли інструмент перетворюється на «чорну скриньку».

### **1.2. Анатомія кризи: Феномен «непроникності» (Case Study: Issue \#376)**

Критичним моментом для розуміння поточної ситуації з Beads є аналіз дискусії в issue \#376 на GitHub.3 Цей артефакт є не просто скаргою користувача, а детальним діагнозом системних проблем у комунікації між розробниками інструменту та його користувачами (як людьми, так і машинами).

#### **1.2.1. Ерозія технічної довіри через «маркетинговий шум»**

Користувач srobertson вказує на «bold, overhyped claims», які характерні для тексту, згенерованого LLM (зокрема Claude).3 Це явище можна назвати «галюцинацією успіху». Мовні моделі, навчені на маркетингових матеріалах, схильні перебільшувати можливості та згладжувати кути. В контексті інженерного інструменту це катастрофічно. Коли документація стверджує, що інструмент працює «безшовно» (seamlessly), а інженер стикається з конфліктом злиття (merge conflict) бази даних на першому ж кроці, довіра руйнується миттєво.  
Для проекту рівня Beads, який претендує на роль фундаментальної інфраструктури (як Git), стиль документації повинен бути максимально сухим, фактологічним та обережним. Кожне твердження про «автоматичне вирішення конфліктів» повинно супроводжуватися виноскою про граничні умови, де це не спрацює.

#### **1.2.2. Відсутність архітектурної прозорості**

Найбільш серйозним звинуваченням є «непроникність» (impenetrability) системи через відсутність пояснення взаємодії компонентів: JSON, SQL та Git.3 Beads використовує складну архітектуру:

* **JSONL** (у папці .beads): Канонічне джерело даних, що версіонується Git.  
* **SQLite** (локальний файл .db): Кеш для швидкого доступу та побудови графів.  
* Daemon: Процес, що синхронізує ці два стани.  
  Без чіткої діаграми потоків даних (Data Flow Diagram), користувач не розуміє, що є первинним. Якщо видалити файл .db, чи втратяться дані? Якщо відредагувати JSON вручну, чи підхопить зміни SQLite? Відсутність відповідей на ці питання робить інструмент небезпечним для використання в продакшн-середовищі.

#### **1.2.3. Рекурсія агентної помилки**

Унікальним спостереженням є те, що «агенти реагують на проблеми в репозиторії».3 Це створює замкнене коло: агент читає погану документацію \-\> робить неправильну дію \-\> отримує помилку \-\> намагається виправити її, базуючись на тій же поганій документації. Це призводить до ентропійного вибуху в репозиторії (створення дублікатів баз даних, нескінченні цикли в графі задач).  
Це підкреслює необхідність існування «Золотого Стандарту» документації, написаного людиною, який виступає як арбітр істини (ground truth) для розриву таких рекурсивних петель.

**Розділ 2\. Подвійна природа сприйняття: Люди проти Машин**

Завдання створення єдиного сайту документації для Beads вимагає задоволення двох радикально відмінних моделей споживання інформації. Ми не можемо просто «додати AI-плагін»; ми повинні фундаментально переосмислити структуру інформації.

### **Таблиця 1\. Порівняльний аналіз вимог до документації**

| Характеристика | Людське сприйняття (Human Cognition) | Машинне сприйняття (LLM Context) |
| :---- | :---- | :---- |

| **Контекст** | Потребує поступового занурення (від простого до складного). | Потребує повного контексту (full context) або ефективного пошуку (RAG). |
| **Обмеження** | Когнітивне навантаження, втома, втрата уваги. | Обмеження вікна контексту (token limit), вартість обробки. |

| **Стиль** | Наративний, педагогічний (Diátaxis). | Імперативний, декларативний, лаконічний. |

### **2.1. Стандарт llms.txt: Інженерія контексту для AI**

Специфікація /llms.txt є новим стандартом де\-факто для веб\-сайтів, які прагнуть бути дружніми до LLM.4 Вона виникла з розуміння того, що HTML-сторінки, насичені JavaScript, CSS та навігаційними елементами, є «шумом» для мовних моделей. Парсинг такого контенту є неефективним та дорогим.

#### **2.1.1. Архітектура файлів стандарту**

Для Beads реалізація повинна включати два ключові компоненти в корені сайту документації:

1. /llms.txt (Навігаційний маніфест):  
   Цей файл діє як «карта місцевості» для агента. Він повинен бути лаконічним (щоб поміститися навіть у найменше контекстне вікно) і містити посилання на найбільш релевантні розділи документації.5  
   * **Структура:**  
     * Заголовок H1 з назвою проекту.  
     * Blockquote з мета-описом ("Beads is a distributed graph issue tracker...").  
     * Секції H2 (Core, API, Concepts) зі списками посилань.  
   * **Призначення:** Дозволяє агенту вирішити, який саме документ йому потрібно завантажити для відповіді на конкретне питання, реалізуючи таким чином агента з активним пошуком (Active Search Agent).  
2. /llms-full.txt (Консолідоване знання):  
   Це «святий грааль» для автономних агентів. Це файл, що містить весь корисний контент документації, зшитий в один суцільний Markdown-документ.6  
   * **Перевага:** Сучасні моделі (Claude 3.5 Sonnet, GPT-4o) мають вікна контексту 128k-200k токенів. Завантаження всієї документації Beads (яка навряд чи перевищує 50k токенів) в один промпт дозволяє моделі мати «глобальне бачення» проекту. Це критично для виявлення складних залежностей, які можуть бути розкидані по різних сторінках (наприклад, зв'язок між конфігурацією Git hooks та поведінкою демона).

#### **2.1.2. Оптимізація контенту для LLM**

Просте копіювання Markdown-файлів недостатнє. Для llms-full.txt необхідно провести пре-процесинг:

* **Очищення:** Видалення всіх елементів, що не несуть семантичного навантаження для коду (емодзі, маркетингові вставки, заклики підписатися на розсилку).  
* **Розгортання сніпетів:** Якщо в людській документації код може бути скороченим для читабельності, для LLM він повинен бути повним і валідним. Агент навчається на прикладах (few-shot learning); якщо приклад неповний, агент згенерує неповний код.8  
* **Explicit Schema Definition:** Включення повних JSON схем для всіх типів об'єктів Beads. Це дозволяє агенту валідувати свої вихідні дані (output validation) перед тим, як намагатися зберегти їх у файл.

### **2.2. Фреймворк Diátaxis: Структурування для людей**

Для вирішення проблеми людського сприйняття ми звертаємося до фреймворку **Diátaxis**, який пропонує науково обґрунтований підхід до організації технічної документації. Він базується на розподілі контенту за двома осями: **Теорія/Практика** та **Навчання/Робота**.9

#### **Квадрант 1: Tutorials (Уроки)**

* *Фокус:* Навчання через дію.  
* *Для Beads:* Покроковий сценарій "Перші 10 хвилин з Beads". Користувач повинен встановити CLI, ініціалізувати репозиторій і створити першу задачу.  
* *Критично:* Туторіал повинен гарантувати успіх. Жодних розгалужень, жодних складних рішень. Тільки прямий шлях до результату.

#### **Квадрант 2: How-to Guides (Практичні інструкції)**

* *Фокус:* Вирішення реальних проблем.  
* *Для Beads:* Це відповідь на критику \#376. Нам потрібні гайди для конкретних сценаріїв:  
  * "Як відновити базу даних після збою".  
  * "Як налаштувати Beads для команди з 5 осіб".  
  * "Як інтегрувати Beads у CI/CD пайплайн".  
* *Особливість:* Ці гайди припускають, що користувач вже знає основи, і фокусуються на конкретній цілі.

#### **Квадрант 3: Reference (Довідка)**

* *Фокус:* Опис фактів.  
* *Для Beads:*  
  * **CLI Command Reference:** Сухий, вичерпний опис кожної команди, аргументу та прапора. Це ідеальний матеріал для автоматичного включення в llms.txt.  
  * **Configuration Reference:** Опис усіх параметрів .beads/config.

#### **Квадрант 4: Explanation (Пояснення/Розуміння)**

* *Фокус:* Побудова ментальної моделі.  
* *Для Beads:* Це «відсутня ланка». Тут має бути описана архітектура, філософія "Graph over List", механіка синхронізації та модель даних. Без цього розділу користувач не зможе зрозуміти *чому* система поводиться саме так, а не інакше.

**Розділ 3\. Архітектурна реконструкція: Відновлення втраченого знання (Data Model)**

Цей розділ звіту виконує роль зразка того контенту, який **зобов'язаний** з'явитися в документації Beads для закриття issue \#376. Базуючись на фрагментарних відомостях з дослідження 2, ми реконструюємо архітектуру системи.

### **3.1. Тріада Даних: Git, SQLite, JSONL**

Архітектура Beads базується на взаємодії трьох компонентів, кожен з яких відіграє специфічну роль у забезпеченні консистентності та доступності даних.

#### **3.1.1. JSONL (Persistent Storage Layer)**

* **Розташування:** .beads/issues.jsonl (та інші файли в цій директорії).  
* **Роль:** Канонічне джерело істини (Source of Truth).  
* **Механіка:** Дані зберігаються у форматі JSON Lines, де кожен рядок є валідним JSON-об'єктом, що представляє собою задачу (issue) або коментар.  
* **Зв'язок з Git:** Оскільки це текстові файли, вони ідеально підходять для версіонування у Git. Git забезпечує історію змін, розгалуження (branching) та злиття (merging). Коли ви перемикаєте гілку в Git (git checkout), ви автоматично перемикаєте контекст задач Beads. Це фундаментальна перевага перед Jira, де задачі існують поза контекстом коду.

#### **3.1.2. SQLite (Query & Caching Layer)**

* **Розташування:** Локальний файл, зазвичай \~/.beads/default.db або локальний .beads/beads.db (залежно від налаштувань init).11  
* **Роль:** Високопродуктивний індекс.  
* **Механіка:** JSONL файли не підходять для складних запитів (наприклад, "знайти всі незаблоковані задачі з пріоритетом High"). Тому Beads імпортує дані з JSONL у реляційну базу даних SQLite.  
* **Ефемерність:** Документація повинна чітко стверджувати: **База даних SQLite є похідною**. Її можна безпечно видалити, і вона буде відновлена (rehydrated) з JSONL файлів командою bd sync. Це знання критично важливе для аварійного відновлення.

#### **3.1.3. Daemon (Synchronization Engine)**

* **Роль:** Посередник між файловою системою та базою даних.  
* **Механіка:** Демон використовує механізми спостереження за файловою системою (fsnotify).  
  * *Шлях 1 (Write):* Коли користувач/агент виконує bd create, запис йде в БД, а потім демон скидає його в JSONL.  
  * *Шлях 2 (Read):* Коли Git оновлює файли (після git pull), демон детектує зміни і оновлює БД.  
* **Точка відмови:** Якщо демон не запущений, стан БД і файлів розсинхронізується. Це пояснює багато проблем, описаних у issue \#376.

### **3.2. Модель даних C4 (Visualizing the Architecture)**

Для візуалізації цієї архітектури в документації (розділ Explanation) необхідно використати діаграми C4.14

**Container Diagram (Опис для Mermaid.js):**

* **User/Agent:** Взаємодіє через CLI.  
* **Beads CLI:** Приймає команди, валідує вхідні дані.  
* **SQLite DB:** Зберігає граф залежностей, вектори (embeddings) для семантичного пошуку.  
* **Sync Service (Daemon):** Двосторонній потік даних між DB та файловою системою.  
* **Git:** Зовнішня система, що керує версіями файлів .beads/\*.jsonl.

Ця візуалізація дає інженерам миттєве розуміння того, де шукати проблему. Якщо дані є в файлі, але немає в bd list \-\> проблема в демоні або БД. Якщо даних немає в файлі \-\> проблема в збереженні або Git.

**Розділ 4\. Технічна імплементація на базі Docusaurus**

Створення єдиного сайту, що обслуговує і людей, і машин, вимагає правильного налаштування платформи Docusaurus.

### **4.1. Інтеграція плагіна docusaurus-plugin-llms-txt**

Згідно з дослідженими матеріалами 7, цей плагін є оптимальним рішенням. Однак, «коробкове» налаштування потребує адаптації.

### **4.3. Автоматизація та CI/CD**

Генерація llms.txt повинна бути частиною білд-процесу.

* **Build Step:** npm run build (включає генерацію плагіном).  
* **Validation:** Додати крок у CI, який перевіряє валідність згенерованого llms.txt (чи є він доступним, чи не пустий, чи не перевищує розумні ліміти розміру).  
* **Versioning:** При релізі нової версії Beads, документація повинна архівуватися. Docusaurus підтримує версіонування. Важливо, щоб llms.txt також був версіонованим (наприклад, /v0.37/llms.txt), щоб агенти, які працюють зі старішими версіями софту, отримували відповідну документацію.

**Розділ 5\. Операційні карти (Runbooks) та відновлення після збоїв**

Цей розділ є прямою відповіддю на потребу в "Human Intervention".3 Runbooks (операційні карти) — це чіткі інструкції для вирішення інцидентів, які повинні бути зрозумілі втомленому інженеру о 3-й годині ночі.17

### **5.1. Runbook: Відновлення пошкодженої бази даних (Database Corruption)**

**Симптоми:**

* CLI видає помилку "database disk image is malformed".  
* Агенти повідомляють про неможливість запису.  
* Команда bd list повертає пустий список, хоча файли .beads/\*.jsonl існують.

**Діагностика:**

1. Перевірити цілісність файлу SQLite:  
   Bash  
   sqlite3 \~/.beads/default.db "PRAGMA integrity\_check;"

2. Якщо результат не "ok", база пошкоджена.19

Процедура відновлення (The Fix):  
Оскільки SQLite є кешем, ми можемо його безпечно перестворити.

1. **Зупинити демона:**  
   Bash  
   pkill bd-daemon

2. **Зробити бекап поточної БД (для форензики):**  
   Bash  
   mv \~/.beads/default.db \~/.beads/default.db.bak

3. Примусова регідратація:  
   Запустити команду синхронізації, яка зчитає всі JSONL файли та побудує нову базу.  
   Bash  
   bd sync \--force-rebuild

4. Перевірка:  
   Виконати bd list, щоб переконатися, що задачі з'явилися.

### **5.2. Runbook: Вирішення циклічних залежностей (Circular Dependencies)**

**Симптоми:**

* Агент входить у нескінченний цикл при спробі визначити наступну задачу.  
* Граф задач не відображається.

Контекст:  
Агенти можуть помилково створити цикл: Задача А блокує В, В блокує С, С блокує А. Це ламає логіку топологічного сортування.  
**Процедура:**

1. Експорт графа:  
   Використати CLI для експорту залежностей у формат, зрозумілий людині.  
   Bash  
   bd export \--format=dot \> graph.dot

2. Візуалізація (опціонально):  
   Використати Graphviz для перегляду graph.dot і візуального пошуку циклу.  
3. Розрив циклу:  
   Відредагувати JSONL файл проблемної задачі вручну (видалити parent\_hash або blocks поле), або використати команду:  
   Bash  
   bd update \<issue-id\> \--remove-blocker \<blocker-id\>

4. Синхронізація:  
   Виконати bd sync для оновлення графа.

### **5.3. Runbook: Глибоке очищення (Nuclear Option)**

У випадку повної втрати контролю над станом (наприклад, агентна рекурсія створила тисячі файлів), потрібна процедура скидання.

1. Git Reset:  
   Використати Git для повернення до останнього стабільного коміту.  
   Bash  
   git checkout \<stable-commit-hash\> \--.beads/

2. Очищення БД:  
   Видалити локальну базу даних.  
3. Re-init:  
   Виконати bd init, щоб налаштувати оточення з чистого листа, базуючись на відновлених файлах.

Ці інструкції повинні бути виділені в окремий розділ документації "Disaster Recovery" і мати тег "High Priority" для індексації в llms.txt.
